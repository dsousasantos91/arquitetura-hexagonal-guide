O que é?

Arquitetura Hexagonal

Proposta em 2005 por Alistair Cockburn, a Arquitetura Hexagonal — também chamada de Ports & Adapters — é um padrão de design de software que organiza o sistema em torno de seu núcleo de negócios (domínio), isolando-o completamente do mundo externo.

A ideia central é: o domínio não deve saber quem está falando com ele. Não importa se quem envia um comando é uma API REST, uma fila de mensagens, um CLI ou um teste automatizado — o domínio recebe, processa e responde através de interfaces bem definidas chamadas portas (ports).

💡 A Metáfora do Hexágono

O hexágono não representa exatamente seis lados. Ele simboliza que o núcleo da aplicação pode ter múltiplos pontos de entrada e saída, todos tratados da mesma forma por meio de ports e adapters.

Os três elementos fundamentais

Domain (Núcleo) O coração da aplicação. Contém entidades, regras de negócio e casos de uso. Não depende de nada externo — nem de banco de dados, nem de HTTP, nem de frameworks.
Ports (Portas) Interfaces (contratos) que definem como o mundo externo pode conversar com o domínio (driving ports) e como o domínio conversa com o mundo externo (driven ports).
Adapters (Adaptadores) Implementações concretas das portas. Traduzem o mundo externo (HTTP, SQL, arquivos) para a linguagem do domínio e vice-versa.
⬡ Driving vs Driven

Driving adapters (esquerda) acionam o domínio: REST controllers, CLIs, consumidores de mensagem. Driven adapters (direita) são acionados pelo domínio: repositórios de banco, serviços de e-mail, filas.

02 / 04
Para que serve?

Por que adotar essa arquitetura?

O problema que a Arquitetura Hexagonal resolve é clássico: à medida que um sistema cresce, o código de negócio fica enredado com detalhes técnicos — queries SQL misturadas com lógica de validação, chamadas HTTP dentro de serviços de domínio, testes que precisam de banco de dados rodando. O resultado é um sistema frágil, difícil de testar e impossível de evoluir.

🧪

Testabilidade radical

Como o domínio não depende de nada, você testa toda a regra de negócio com mocks simples, sem banco de dados, sem servidor HTTP rodando.

🔄

Troca de infraestrutura

Trocar PostgreSQL por MongoDB? Migrar de REST para gRPC? Basta criar um novo adapter. O domínio não precisa nem saber que mudou.

🏗️

Independência de framework

O núcleo do seu negócio não é escravo do Spring. O Spring Boot, Quarkus ou Micronaut é só mais um detalhe na borda — um adapter que poderia ser trocado sem tocar no domínio.

📐

Regra de dependência clara

As dependências sempre apontam para dentro. O domínio jamais importa algo da infraestrutura. A arquitetura impõe isso estruturalmente.

🚀

Múltiplas entradas

A mesma regra de negócio pode ser acionada por uma API REST, uma fila de mensagens ou um cron job, sem duplicação de código.

📖

Legibilidade e intenção

Ler o domínio é ler o negócio. Não há SQL, não há JSON, não há HTTP. Código que conta a história da empresa.

Comparação: código acoplado vs. hexagonal

Cenário Código acoplado Hexagonal
Testar regra de negócio Precisa subir banco + servidor ✓ Só instanciar o use case
Trocar banco de dados Refatorar metade do código ✓ Criar novo Repository adapter
Adicionar canal (CLI, fila) Copiar/colar lógica de negócio ✓ Novo driving adapter
Trocar framework web Reescrever a aplicação ✓ Novo HTTP adapter
Entender o negócio pelo código ✗ Lógica espalhada ✓ Domain é autoexplicativo
⚠️ Quando NÃO usar

Hexagonal adiciona camadas e indireção. Para microsserviços pequenos, scripts, provas de conceito ou CRUDs simples, pode ser overhead desnecessário. Use quando a complexidade do domínio justifica o investimento.

Passo a passo de implementação

01

Modele as entidades do domínio

Comece pelas entidades e value objects. Eles representam conceitos do negócio, têm identidade ou valor e não têm nenhuma dependência externa.

02

Defina os Ports de saída (out)

O use case precisa de dados externos? Defina uma interface (Port OUT) que descreva o contrato, sem mencionar tecnologia alguma.

03

Implemente os Use Cases

O use case recebe os ports por injeção de dependência e orquestra a lógica. Ele só conhece interfaces, nunca implementações.

04

Defina os Ports de entrada (in)

Crie interfaces que expõem as operações do domínio para o mundo externo. Os use cases implementam essas interfaces.

05

Implemente os Adapters

Agora implemente as infraestruturas concretas: o repository que faz SQL, o controller HTTP, o consumidor de fila, etc.

06

Configure o container Spring (DI)

Na camada de infraestrutura, conecte tudo via @Configuration e @Bean: injete os adapters concretos nos use cases e exponha os beans para o Spring gerenciar.

Código: Port de Saída (interface)

// domain/port/out/OrderRepositoryPort.java
// Interface pura — sem import de JPA, Spring ou qualquer framework

package com.empresa.pedidos.domain.port.out;

public interface OrderRepositoryPort {
    void save(Order order);
    Optional<Order> findById(String id);
    List<Order> findByCustomerId(String customerId);
}port / out

Código: Use Case do domínio

// domain/service/CreateOrderService.java
// @Component fica aqui — mas sem import de JPA, HTTP ou qualquer infra

@Component
public class CreateOrderService implements CreateOrderUseCase {

    private final OrderRepositoryPort    orderRepository;
    private final CustomerRepositoryPort customerRepository;
    private final NotificationPort       notification;

    public CreateOrderService(
        OrderRepositoryPort    orderRepository,
        CustomerRepositoryPort customerRepository,
        NotificationPort       notification
    ) {
        this.orderRepository    = orderRepository;
        this.customerRepository = customerRepository;
        this.notification       = notification;
    }

    @Override
    public CreateOrderOutput execute(CreateOrderCommand command) {
        // Busca o cliente (port, não sabe qual banco)
        Customer customer = customerRepository.findById(command.customerId())
            .orElseThrow(() -> new DomainException("Cliente não encontrado."));

        // Cria e valida (regra de negócio pura)
        Order order = Order.create(command.customerId(), command.items());

        // Persiste via port (não sabe que é JPA)
        orderRepository.save(order);

        // Notifica via port (não sabe que é Mailjet)
        notification.sendOrderConfirmation(order, customer.email());

        return new CreateOrderOutput(order.getId(), order.getTotal(), order.getStatus());
    }
}use case

Código: Adapter de saída (JPA Repository)

// adapter/out/persistence/JpaOrderRepository.java
// Implementa o port. Aqui sim usamos JPA, Spring Data, o que precisar.

@Component
public class JpaOrderRepository implements OrderRepositoryPort {

    private final SpringDataOrderRepository springRepo;

    public JpaOrderRepository(SpringDataOrderRepository springRepo) {
        this.springRepo = springRepo;
    }

    @Override
    public void save(Order order) {
        springRepo.save(OrderEntity.fromDomain(order));
    }

    @Override
    public Optional<Order> findById(String id) {
        return springRepo.findById(id)
            .map(OrderEntity::toDomain);
    }
}

interface SpringDataOrderRepository extends JpaRepository<OrderEntity, String> {}adapter / out
🔑 A regra de ouro

Se você está escrevendo código em domain/ e sente vontade de importar algo de adapter/ ou de um pacote de infraestrutura (JPA, Hibernate, Spring Data, RestTemplate), pare. Isso é um sinal de que você rompeu a arquitetura. Crie um port.

Exemplo real

Sistema de pedidos em Java + Spring Boot

Vamos construir um sistema de criação de pedidos completo com Arquitetura Hexagonal em Java 21 + Spring Boot 3. O sistema aceita pedidos via HTTP, persiste com JPA e envia um e-mail de confirmação.

📦 Contexto do sistema

Uma loja online precisa de um endpoint POST /orders que: valide o pedido, calcule o total, salve via JPA (PostgreSQL) e notifique o cliente por e-mail. Deve ser testável sem banco — apenas com JUnit 5 + Mockito — e facilmente migrável para outro banco no futuro.

1. A Entidade de Domínio

// domain/entity/Order.java
// Plain Java — nenhuma anotação JPA, nenhum import do Spring

public class Order {

    private final String        id;
    private final String        customerId;
    private final List<OrderItem> items;
    private final BigDecimal    total;
    private       OrderStatus   status;
    private final LocalDateTime createdAt;

    private Order(String id, String customerId, List<OrderItem> items,
                  BigDecimal total, OrderStatus status, LocalDateTime createdAt) {
        this.id         = id;
        this.customerId = customerId;
        this.items      = Collections.unmodifiableList(items);
        this.total      = total;
        this.status     = status;
        this.createdAt  = createdAt;
    }

    // Factory method — encapsula as regras de criação
    public static Order create(String customerId, List<OrderItem> items) {
        if (items == null || items.isEmpty()) {
            throw new DomainException("Um pedido precisa ter ao menos um item.");
        }
        BigDecimal total = items.stream()
            .map(i -> i.price().multiply(BigDecimal.valueOf(i.quantity())))
            .reduce(BigDecimal.ZERO, BigDecimal::add);

        if (total.compareTo(BigDecimal.ZERO) <= 0) {
            throw new DomainException("Total do pedido deve ser positivo.");
        }
        return new Order(UUID.randomUUID().toString(), customerId,
                         items, total, OrderStatus.PENDING, LocalDateTime.now());
    }

    public void confirm() {
        if (this.status != OrderStatus.PENDING) {
            throw new DomainException("Apenas pedidos pendentes podem ser confirmados.");
        }
        this.status = OrderStatus.CONFIRMED;
    }

    // getters...
}entity

3. O Use Case (Domain Service)

// domain/service/CreateOrderService.java

@Component
public class CreateOrderService implements CreateOrderUseCase {

    private final OrderRepositoryPort    orderRepository;
    private final CustomerRepositoryPort customerRepository;
    private final NotificationPort       notification;

    public CreateOrderService(
        OrderRepositoryPort    orderRepository,
        CustomerRepositoryPort customerRepository,
        NotificationPort       notification
    ) {
        this.orderRepository    = orderRepository;
        this.customerRepository = customerRepository;
        this.notification       = notification;
    }

    @Override
    public CreateOrderOutput execute(CreateOrderCommand command) {
        // Busca cliente (port OUT, não sabe qual banco)
        Customer customer = customerRepository.findById(command.customerId())
            .orElseThrow(() -> new DomainException("Cliente não encontrado."));

        // Cria entidade com validação (regra de negócio pura)
        Order order = Order.create(command.customerId(), command.items());

        // Persiste via port OUT
        orderRepository.save(order);

        // Notifica via port OUT
        notification.sendOrderConfirmation(order, customer.email());

        return new CreateOrderOutput(order.getId(), order.getTotal(), order.getStatus());
    }
}use case

4. Adapters de Saída (Driven) — o mesmo Port, implementações diferentes

O OrderRepositoryPort é a interface imutável do domínio. Qualquer classe que a implemente pode ser injetada — basta trocar o bean no @Configuration. O domínio e o use case não mudam uma linha.

🔁 Como a troca acontece na prática

Quando há apenas uma classe implementando cada port e ela está anotada com @Component, o Spring resolve automaticamente. Quando há múltiplas implementações do mesmo port (ex: JpaOrderRepository e MongoOrderRepository ambas no classpath), você precisa ser explícito — seja via @Primary, @Profile, ou wiring manual no @Configuration. Em todos os casos, o domínio e o use case não mudam uma linha.

● Opção A — Spring Data JPA (PostgreSQL) 
// adapter/out/persistence/JpaOrderRepository.java

@Component
public class JpaOrderRepository implements OrderRepositoryPort {

    private final SpringDataOrderRepository springRepo;

    public JpaOrderRepository(SpringDataOrderRepository springRepo) {
        this.springRepo = springRepo;
    }

    @Override
    public void save(Order order) {
        springRepo.save(OrderEntity.fromDomain(order));
    }

    @Override
    public Optional<Order> findById(String id) {
        return springRepo.findById(id).map(OrderEntity::toDomain);
    }
}

interface SpringDataOrderRepository
    extends JpaRepository<OrderEntity, String> {}adapter / out · JPA
● Opção B — Spring Data MongoDB 
// adapter/out/persistence/MongoOrderRepository.java
// Mesma interface, tecnologia completamente diferente.
// O domínio não sabe — e não precisa saber — desta classe.

@Component
public class MongoOrderRepository implements OrderRepositoryPort {

    private final MongoTemplate mongoTemplate;

    public MongoOrderRepository(MongoTemplate mongoTemplate) {
        this.mongoTemplate = mongoTemplate;
    }

    @Override
    public void save(Order order) {
        mongoTemplate.save(OrderDocument.fromDomain(order), "orders");
    }

    @Override
    public Optional<Order> findById(String id) {
        OrderDocument doc = mongoTemplate.findById(id, OrderDocument.class, "orders");
        return Optional.ofNullable(doc).map(OrderDocument::toDomain);
    }
}adapter / out · MongoDB
● Opção C — NotificationPort via Mailjet 
// adapter/out/notification/MailjetNotificationAdapter.java

@Component
public class MailjetNotificationAdapter implements NotificationPort {

    private final MailjetClient client;

    public MailjetNotificationAdapter(MailjetClient client) {
        this.client = client;
    }

    @Override
    public void sendOrderConfirmation(Order order, String email) {
        MailjetRequest request = new MailjetRequest(Emailv31.resource)
            .property(Emailv31.MESSAGES, new JSONArray().put(
                new JSONObject()
                    .put("To", new JSONArray().put(new JSONObject().put("Email", email)))
                    .put("Subject", "Pedido " + order.getId() + " confirmado")
                    .put("TextPart", "Total: R$ " + order.getTotal())
            ));
        client.post(request);
    }
}adapter / out · Mailjet
● Opção D — NotificationPort via Amazon SES 
// adapter/out/notification/SesNotificationAdapter.java
// Troca Mailjet por AWS SES — zero alteração no domínio.

@Component
public class SesNotificationAdapter implements NotificationPort {

    private final SesClient sesClient;

    public SesNotificationAdapter(SesClient sesClient) {
        this.sesClient = sesClient;
    }

    @Override
    public void sendOrderConfirmation(Order order, String email) {
        sesClient.sendEmail(r -> r
            .destination(d -> d.toAddresses(email))
            .message(m -> m
                .subject(s -> s.data("Pedido " + order.getId() + " confirmado"))
                .body(b -> b.text(t -> t.data("Total: R$ " + order.getTotal())))
            )
            .source("noreply@loja.com.br")
        );
    }
}adapter / out · AWS SES

5. Adapters de Entrada (Driving) — o mesmo Use Case, acionado de formas diferentes

O CreateOrderUseCase é a interface que o domínio expõe. Qualquer adapter que a receba por injeção pode acionar o mesmo fluxo de negócio — seja via HTTP, fila de mensagens ou linha de comando. O use case não muda.

● Opção A — REST Controller (Spring MVC) 
// adapter/in/web/OrderController.java
// Aciona o domínio via chamada HTTP POST /orders

@RestController
@RequestMapping("/orders")
public class OrderController {

    private final CreateOrderUseCase createOrderUseCase;

    public OrderController(CreateOrderUseCase createOrderUseCase) {
        this.createOrderUseCase = createOrderUseCase;
    }

    @PostMapping
    @ResponseStatus(HttpStatus.CREATED)
    public CreateOrderOutput createOrder(
        @RequestBody @Valid CreateOrderRequest request
    ) {
        return createOrderUseCase.execute(
            new CreateOrderCommand(request.customerId(), request.items())
        );
    }

    @ExceptionHandler(DomainException.class)
    @ResponseStatus(HttpStatus.UNPROCESSABLE_ENTITY)
    public ErrorResponse handleDomainException(DomainException ex) {
        return new ErrorResponse(ex.getMessage());
    }
}adapter / in · HTTP
● Opção B — Kafka Consumer (Spring Kafka) 
// adapter/in/queue/OrderKafkaConsumer.java
// O mesmo use case, agora acionado por uma mensagem no tópico Kafka.
// O domínio não sabe de onde vem o comando.

@Component
public class OrderKafkaConsumer {

    private final CreateOrderUseCase createOrderUseCase;

    public OrderKafkaConsumer(CreateOrderUseCase createOrderUseCase) {
        this.createOrderUseCase = createOrderUseCase;
    }

    @KafkaListener(topics = "orders.create", groupId = "order-service")
    public void consume(CreateOrderMessage message) {
        CreateOrderCommand command = new CreateOrderCommand(
            message.customerId(),
            message.items()
        );
        createOrderUseCase.execute(command);
    }
}adapter / in · Kafka
● Opção C — CLI (Spring Shell) 
// adapter/in/cli/OrderShellAdapter.java
// O mesmo use case, agora acionado via terminal.
// Útil para scripts internos, migrations, ferramentas de suporte.

@ShellComponent
public class OrderShellAdapter {

    private final CreateOrderUseCase createOrderUseCase;

    public OrderShellAdapter(CreateOrderUseCase createOrderUseCase) {
        this.createOrderUseCase = createOrderUseCase;
    }

    @ShellMethod(key = "create-order", value = "Cria um pedido via CLI")
    public String createOrder(
        @ShellOption String customerId,
        @ShellOption String productId,
        @ShellOption int    quantity
    ) {
        CreateOrderCommand command = new CreateOrderCommand(
            customerId,
            List.of(new OrderItem(productId, BigDecimal.TEN, quantity))
        );
        CreateOrderOutput output = createOrderUseCase.execute(command);
        return "Pedido criado: " + output.orderId() + " | Total: R$ " + output.total();
    }
}adapter / in · CLI
● Opção D — Scheduled Job (Spring @Scheduled) 
// adapter/in/scheduler/RecurringOrderScheduler.java
// O mesmo use case, agora acionado automaticamente a cada hora.
// Pedidos recorrentes sem alterar nenhuma regra de negócio.

@Component
public class RecurringOrderScheduler {

    private final CreateOrderUseCase       createOrderUseCase;
    private final RecurringOrderQueryPort   recurringOrders;

    public RecurringOrderScheduler(
        CreateOrderUseCase     createOrderUseCase,
        RecurringOrderQueryPort recurringOrders
    ) {
        this.createOrderUseCase = createOrderUseCase;
        this.recurringOrders    = recurringOrders;
    }

    @Scheduled(cron = "0 0 * * * *") // toda hora
    public void processRecurringOrders() {
        recurringOrders.findDue().forEach(recurring ->
            createOrderUseCase.execute(
                new CreateOrderCommand(recurring.customerId(), recurring.items())
            )
        );
    }
}adapter / in · Scheduler
💡 O ponto central desta seção

Quatro adapters de entrada diferentes (HTTP, Kafka, CLI, Scheduler) e quatro de saída diferentes (JPA, MongoDB, Mailjet, SES) — todos compartilhando exatamente o mesmo domínio e use case, sem nenhuma alteração. Isso é o que a Arquitetura Hexagonal garante: o contrato (port) é estável, a implementação (adapter) é intercambiável.

6. Configuração do Container Spring (DI) — como trocar implementações

Existem três formas práticas de controlar qual adapter o Spring injeta. Escolha a que melhor se encaixa no seu contexto.

Estratégia 1 — Um único @Component (caso mais comum)
// Se só existe UMA classe implementando o port no classpath,
// o Spring resolve automaticamente — sem configuração extra.

@Component  // ← Spring descobre e injeta esta implementação sozinho
public class JpaOrderRepository implements OrderRepositoryPort { ... }

// Para trocar para Mongo: remova @Component de JpaOrderRepository
// e adicione @Component em MongoOrderRepository. Fim.
@Component
public class MongoOrderRepository implements OrderRepositoryPort { ... }auto-wiring
Estratégia 2 — @Profile (ambientes diferentes)
// Ideal quando você quer JPA em produção e Mongo em staging,
// ou um adapter in-memory nos testes.

@Component
@Profile("prod")
public class JpaOrderRepository implements OrderRepositoryPort { ... }

@Component
@Profile("staging")
public class MongoOrderRepository implements OrderRepositoryPort { ... }

@Component
@Profile("test")
public class InMemoryOrderRepository implements OrderRepositoryPort {
    private final Map<String, Order> store = new HashMap<>();

    @Override
    public void save(Order order) { store.put(order.getId(), order); }

    @Override
    public Optional<Order> findById(String id) {
        return Optional.ofNullable(store.get(id));
    }
}

// Ativa via: spring.profiles.active=prod | staging | test@Profile
Estratégia 3 — @Configuration com wiring explícito (máximo controle)
// infrastructure/config/BeanConfiguration.java
// Você nomeia explicitamente qual classe concreta resolve cada port.
// Nenhum adapter precisa de @Component — o @Bean substitui.

@Configuration
public class BeanConfiguration {

    // ── Driven adapters (OUT) — você escolhe a implementação aqui ──────

    @Bean
    public OrderRepositoryPort orderRepository(SpringDataOrderRepository springRepo) {
        return new JpaOrderRepository(springRepo);
        // ↑ Trocar para Mongo: return new MongoOrderRepository(mongoTemplate);
    }

    @Bean
    public NotificationPort notification(MailjetClient mailjetClient) {
        return new MailjetNotificationAdapter(mailjetClient);
        // ↑ Trocar para SES: return new SesNotificationAdapter(sesClient);
    }

    // ── Domain service — recebe os ports resolvidos acima ──────────────
    // Note: o @Bean orderRepository() retorna OrderRepositoryPort (interface),
    // não JpaOrderRepository. CreateOrderService só enxerga a interface.

    @Bean
    public CreateOrderUseCase createOrderUseCase(
        OrderRepositoryPort    orderRepository,
        CustomerRepositoryPort customerRepository,
        NotificationPort       notification
    ) {
        return new CreateOrderService(orderRepository, customerRepository, notification);
    }
}explicit wiring

Se você está escrevendo código em domain/ e sente vontade de importar algo de adapter/ ou de um pacote de infraestrutura (JPA, Hibernate, Spring Data, RestTemplate), pare. Isso é um sinal de que você rompeu a arquitetura. Crie um port.

7. Teste unitário — sem banco, sem Spring

// test/domain/service/CreateOrderServiceTest.java
// JUnit 5 + Mockito — zero infraestrutura, roda em milissegundos

@ExtendWith(MockitoExtension.class)
class CreateOrderServiceTest {

SOLID & Patterns

Relação com SOLID e Design Patterns

A Arquitetura Hexagonal não surgiu do nada. Ela é uma composição de princípios e padrões que já existiam — organizados em torno da ideia de que o domínio é o centro e tudo externo fala com ele através de contratos que o próprio domínio define. Entender essas conexões aprofunda o porquê das decisões de design, não apenas o como.

Os cinco princípios SOLID

S
Single Responsibility Principle
Princípio da Responsabilidade Única
Forte
Princípio

Uma classe deve ter apenas um motivo para mudar. Ela deve ser responsável por uma única parte do comportamento do sistema.

Na Hexagonal
OrderController → traduz HTTP
CreateOrderService → orquestra negócio
JpaOrderRepository → persiste dados
MailjetAdapter → envia e-mail

Cada classe tem um único motivo para mudar.
Mudar o banco não toca no controller.
Mudar o e-mail não toca no use case.
O
Open/Closed Principle
Princípio Aberto/Fechado
Forte
Princípio

Entidades de software devem estar abertas para extensão, mas fechadas para modificação. Você estende comportamento adicionando código novo, não alterando o existente.

Na Hexagonal
Precisa de um novo canal de entrada?
→ Crie OrderKafkaConsumer
→ Nada mais muda

Precisa de um novo banco de saída?
→ Crie MongoOrderRepository
→ Nada mais muda

O sistema se estende, nunca se modifica.
L
Liskov Substitution Principle
Princípio da Substituição de Liskov
Forte
Princípio

Objetos de uma subclasse devem poder substituir objetos da superclasse sem quebrar o comportamento do programa. Implementações devem honrar integralmente o contrato da interface.

Na Hexagonal
CreateOrderService recebe:
OrderRepositoryPort

Pode ser injetado:
→ JpaOrderRepository ✓
→ MongoOrderRepository ✓
→ InMemoryOrderRepository ✓

Qualquer implementação é
substituível — o use case não sente.
I
Interface Segregation Principle
Princípio da Segregação de Interfaces
Presente
Princípio

Nenhum cliente deve ser forçado a depender de métodos que não usa. É melhor ter muitas interfaces pequenas e específicas do que uma interface grande e genérica.

Na Hexagonal
✗ Interface única e gorda:
OrderRepository {
save, findById,
findAll, delete,
findByStatus, count... }

✓ Ports segregados por intenção:
CreateOrderPort → só execute()
OrderQueryPort → só findById()
OrderRepositoryPort → save + find
D
Dependency Inversion Principle
Princípio da Inversão de Dependência
Fundação
Princípio

Módulos de alto nível não devem depender de módulos de baixo nível. Ambos devem depender de abstrações. As abstrações não devem depender de detalhes — os detalhes é que dependem das abstrações.

Na Hexagonal
// ✗ Sem inversão (acoplado)
CreateOrderService
→ JpaOrderRepository
→ MailjetAdapter

// ✓ Com inversão (hexagonal)
CreateOrderService
→ OrderRepositoryPort ←
→ NotificationPort ←
↑ implementado por
JpaOrderRepository
MailjetAdapter
🎯 O mais importante dos cinco

O DIP é a fundação da Arquitetura Hexagonal — sem ele, não há ports nem a direção de dependência correta. O ISP é frequentemente citado, mas aparece como consequência natural de ports bem desenhados. Os outros três (SRP, OCP, LSP) emergem automaticamente quando você aplica a arquitetura com disciplina.

Design Patterns relacionados

Além do SOLID, a Arquitetura Hexagonal conversa diretamente com três padrões do Gang of Four e com padrões de domínio de Evans (DDD). Conhecê-los ajuda a entender de onde vêm as decisões de design.

GoF · Estrutural
Adapter
→ Relação direta e literal

O padrão Adapter converte a interface de uma classe em outra que o cliente espera. Os adapters da arquitetura hexagonal são esse padrão: JpaOrderRepository converte a interface JPA para a interface OrderRepositoryPort que o domínio espera. Cockburn não foi acidental ao nomear a arquitetura de Ports & Adapters.

GoF · Comportamental
Strategy
→ Múltiplas implementações de um port

O padrão Strategy define uma família de algoritmos intercambiáveis. Quando você tem JpaOrderRepository, MongoOrderRepository e InMemoryOrderRepository todas implementando o mesmo port, você tem Strategy — e o Spring DI faz o papel do contexto que elege qual usar.

GoF · Estrutural
Facade
→ Use Case como simplificador

O padrão Facade fornece uma interface simplificada para um conjunto complexo de classes. O Use Case age como Facade do domínio: o OrderController não precisa conhecer entidades, value objects ou regras de validação — só chama createOrderUseCase.execute(command) e recebe um resultado.

DDD · Evans
Repository
→ Driven adapters de persistência

O padrão Repository de Evans media entre o domínio e a camada de dados, simulando uma coleção de objetos em memória. Os driven adapters de persistência implementam exatamente isso: o contrato está no vocabulário do domínio (findByCustomerId), a implementação usa JPA, Mongo ou o que for.

GoF · Criacional
Factory Method
→ Criação de entidades de domínio

As entidades de domínio usam factory methods estáticos (Order.create(...)) para encapsular criação e validação, mantendo o construtor privado. Isso impede que objetos inválidos existam — e é um padrão GoF clássico dentro do núcleo da arquitetura.

Enterprise · Fowler
Dependency Injection
→ Wiring de ports e adapters

DI é o mecanismo que permite trocar adapters sem tocar no domínio. O Spring atua como container de DI, resolvendo qual implementação concreta injetar em cada port. Sem DI, você precisaria instanciar tudo manualmente — o que o BeanConfiguration com wiring explícito demonstra.

Mapa de relações resumido

Conceito Tipo Força Manifestação na Hexagonal
DIP SOLID ● Fundação Domínio define ports (abstrações); adapters implementam (detalhes)
SRP SOLID ● Forte Controller só traduz HTTP; Service só orquestra; Repository só persiste
OCP SOLID ● Forte Novos adapters adicionam comportamento sem modificar código existente
LSP SOLID ● Forte JPA, Mongo e InMemory substituem OrderRepositoryPort sem quebrar o use case
ISP SOLID ● Presente Ports pequenos e focados por intenção de uso, não uma interface genérica

Factory Method GoF · Criacional ● Presente Order.create() encapsula criação e validação no núcleo do domínio
💡 Por que isso importa na prática

Reconhecer esses padrões ao revisar código hexagonal tem impacto direto: quando um colega questionar "por que essa interface?", você responde com DIP. Quando perguntar "por que não reutilizar esse adapter?", você responde com SRP. Quando propuser uma interface com 15 métodos, você responde com ISP. A Arquitetura Hexagonal não é um conjunto de regras arbitrárias — ela é a aplicação coerente de princípios que a engenharia de software já provou funcionarem.

Perguntas para autoavaliação

→ O que são Ports e para que servem?
São interfaces (contratos) que definem como o domínio se comunica com o exterior. Ports IN definem operações que o domínio expõe; Ports OUT definem o que o domínio precisa do exterior.
→ Qual é a regra de dependência fundamental?
As dependências sempre apontam para dentro: adapters dependem de ports, ports pertencem ao domínio, o domínio não depende de nada externo.
→ O que é um Driving Adapter vs. Driven Adapter?
Driving adapters acionam o domínio (HTTP controller, CLI, consumer de fila). Driven adapters são acionados pelo domínio (repositório de banco, serviço de e-mail, API externa).
→ Como você testaria o domínio sem banco de dados?
Criando mocks dos ports de saída e injetando-os no use case. Como o domínio depende apenas de interfaces, qualquer objeto que implemente a interface serve como substituto.
→ Se precisar trocar de PostgreSQL para MongoDB, o que muda?
Apenas o adapter de saída (a classe que implementa OrderRepositoryPort). O domínio, os use cases e todos os outros adapters permanecem idênticos.

Conceitos-chave em uma linha

Domain Entidades, value objects, use cases e regras de negócio. Zero dependências externas.
Port IN Interface que expõe uma operação do domínio para o exterior. Implementada pelo use case.
Port OUT Interface que o domínio usa para acessar recursos externos. Implementada pelo adapter.
Driving Adapter Acionador do domínio: HTTP controller, CLI, consumer de mensagem, test runner.
Driven Adapter Acionado pelo domínio: JpaRepository, Redis, Mailjet, S3, Kafka producer.
